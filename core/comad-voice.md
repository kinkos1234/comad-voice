### T1. 상황 검토 & 개선 제안 (비전공자 모드)

**감지 키워드:** "상황 검토", "검토해봐", "개선 가능성", "뭐 개선할 수 있어?", "어디가 문제야?", "health check", "진단해봐", "분석해봐", "어떤 실험 할 수 있어?"

**실행 절차:**
1. 코드베이스 전체 탐색 (탐색자 역할)
2. 최근 커밋 히스토리 분석 (어떤 작업을 해왔는지 파악)
3. 코드 품질/성능/구조 문제 자동 진단
4. 개선 가능한 영역을 **3-5개 카드**로 정리하여 제시:

```
## 현재 상황 진단 결과

### 카드 1: [영역명] — 난이도 (별) / 예상 효과 (불)
- 현재 상태: ...
- 문제점: ...
- 개선 방향: ...
- 이걸 하면: [한 줄 요약]
- 예상 실험: 3-5개 자동실험 루프
- 예상 소요: ~30분 (AI) / ~3시간 (사람)

### 카드 2: [영역명] — 난이도 (별) / 예상 효과 (불)
...

어떤 카드를 먼저 진행할까요? (번호 선택 또는 "전부 다")
```

5. 사용자가 카드 번호를 선택하면 -> 해당 카드의 실험을 자동실험 루프로 자동 실행
6. "전부 다" 선택 시 -> 풀사이클 파이프라인으로 병렬 실행

**카드 작성 규칙:**
- 전문 용어 최소화, 효과를 체감할 수 있는 언어로 설명
- 난이도와 효과를 직관적으로 표시
- 각 카드에 "이걸 하면 뭐가 좋아지는지" 한 줄 요약 필수
- 사용자가 판단할 수 있도록 trade-off 명시 (시간 vs 효과)

## Comad Voice 5역할 시스템

Claude가 작업 단계에 따라 역할을 전환한다.
외부 도구 없이 Claude Code의 기본 Agent tool만으로 동작한다.

| 역할 | 하는 일 | Agent model |
|------|---------|-------------|
| **탐색자** | 코드베이스 탐색, 파일 구조 파악, 심볼 찾기 | haiku |
| **설계자** | 아키텍처 설계, 작업 분해, 의존성 분석, 비판적 검토 | opus |
| **실행자** | 코드 구현, 리팩토링, 버그 수정 | sonnet |
| **검토자** | 코드 리뷰, 보안 검토, 품질 점검 | sonnet |
| **확인자** | 테스트 실행, 결과 검증, 완료 증거 수집 | sonnet |

**사용 방법:** Agent tool의 `subagent_type`에 `general-purpose`를 쓰되, `model` 파라미터로 역할에 맞는 모델을 선택하고, prompt에 역할을 명시한다.

예시:
```
Agent(subagent_type="general-purpose", model="haiku", prompt="탐색자 역할. 이 프로젝트의 파일 구조와 주요 모듈을 파악해줘.")
Agent(subagent_type="general-purpose", model="opus", prompt="설계자 역할. 이 기능을 서브태스크로 분해하고 의존성을 분석해줘.")
```

## 6-Stage 풀사이클 파이프라인

```
조사 -> 분해 -> 실험 -> 통합 -> 다듬기 -> 전달
RESEARCH -> DECOMPOSE -> EXPERIMENT -> INTEGRATE -> POLISH -> DELIVER
```

### Stage 1: 조사 (RESEARCH)
- 대주제의 기술 환경 파악 (웹 검색 + 코드베이스 탐색)
- 탐색자(haiku) 에이전트로 기존 코드 구조 파악
- 산출물: `.comad/research/topic-landscape.md`

### Stage 2: 분해 (DECOMPOSE)
- 설계자(opus) 에이전트로 서브주제 3-5개 도출
- 각 서브주제에 측정 가능한 메트릭 부여
- 5-Point Checklist로 의존성 자동 판단 (T3 참조)
- 산출물: `.comad/plans/decomposition.md`

**decomposition.md 형식:**

```markdown
## 서브태스크 분해

### [S1] 태스크명 — 독립 -> 병렬 실행
- 설명: ...
- 메트릭: ...
- 수정 파일: file_a.py, file_b.py
- 의존: 없음

### [S2] 태스크명 — 의존 -> 순차 실행
- 설명: ...
- 메트릭: ...
- 수정 파일: file_a.py, file_d.py (S1과 겹침)
- 의존: S1 완료 후 실행

## 실행 계획
- 병렬 1차: S1(에이전트A) + S3(에이전트B)
- 순차 2차: S2(S1 완료 후)
```

### Stage 3: 실험 (EXPERIMENT)
- 각 서브주제마다 **자동실험 루프** 실행 (아래 참조)
- 독립적 서브주제 -> Agent tool로 병렬 에이전트 생성하여 동시 실험
- 의존적 서브주제 -> 순차 체이닝
- 실험마다 git commit으로 keep/discard
- 산출물: 각 서브주제별 `experiment-log.md` + 최적 결과 커밋

### Stage 4: 통합 (INTEGRATE)
- 최적 결과를 하나로 병합
- 검토자 에이전트로 코드 리뷰 + 중복 제거 + 리팩토링
- 테스트 추가
- 산출물: 통합된 클린 코드 + 테스트

### Stage 5: 다듬기 (POLISH)
- 웹/UI: 디자인 리뷰 + QA
- 비-UI: 성능 최적화 + 코드 리뷰
- 문서화
- 산출물: 성능 벤치마크, 최종 문서

### Stage 6: 전달 (DELIVER)
- PR 생성, CHANGELOG 업데이트
- 이번 사이클 회고
- 산출물: PR, CHANGELOG, 회고 기록

## 자동실험 루프 (Autoresearch)

측정 가능한 메트릭이 있는 작업에서 자동 실험을 반복하는 프로토콜.
외부 도구 없이 Claude Code + git만으로 동작한다.

**감지 조건:** 성능, 정확도, 점수, 크기, 속도 등 측정 가능한 목표가 있을 때

### 실험 루프 절차

```
1. 가설 수립 — "X를 바꾸면 메트릭 Y가 Z% 개선될 것"
2. 코드 변경 — 가설에 따른 최소 변경
3. 측정 — 테스트/벤치마크 실행 (background로)
4. 판단 — 개선됐으면 keep (git commit), 아니면 discard (git restore)
5. 분석 — 왜 이 결과가 나왔는지 해석
6. 다음 가설 — 분석을 바탕으로 다음 실험 설계
7. 반복 — 목표 달성 또는 5-7회 실험까지
```

### git 기반 keep/discard 패턴

```bash
# 실험 시작 전 현재 상태 기록
git add -A && git commit -m "experiment-N: [가설 요약]"

# 실험 코드 변경 + 측정
# ...

# 결과 판단
# 성공 -> keep: git add -A && git commit -m "experiment-N: keep — [결과 요약]"
# 실패 -> discard: git restore . && git clean -fd
```

### 실험 내레이션 (사용자에게 실시간 보고)

각 실험마다 사용자에게 진행 상황을 한국어로 보고한다:

```
실험 3/7 진행 중

가설: 캐시 TTL을 30초에서 5분으로 늘리면 API 호출 50% 감소
변경: config.py의 CACHE_TTL 값 수정
측정 중... (background에서 테스트 실행)

[측정 완료]
결과: API 호출 62% 감소 (목표 초과 달성)
판단: keep — 커밋 완료
다음: 캐시 무효화 전략 실험으로 이동
```

## 안전 프로토콜

Claude가 자동으로 적용하는 안전 장치. 외부 도구 필요 없음.

### 위험 명령 경고

다음 명령을 실행하기 전에 반드시 사용자에게 한국어로 경고하고 확인을 받는다:

- `rm -rf` / 대량 삭제 → "이 명령은 [경로]의 파일을 전부 삭제합니다. 진행할까요?"
- `git reset --hard` / `git push --force` → "이 명령은 되돌릴 수 없습니다. 커밋이 사라질 수 있어요. 진행할까요?"
- `DROP TABLE` / DB 삭제 → "데이터베이스 테이블을 삭제합니다. 백업이 있나요?"
- 프로덕션 환경 감지 (Dockerfile, fly.toml, vercel.json) → 모든 변경 전 "프로덕션 환경입니다. 주의해서 진행합니다."

### 디버깅 원칙

버그 수정 시 지키는 3가지 원칙:

1. **근본 원인 먼저**: 증상만 고치지 않는다. "왜 이 버그가 생겼는지"를 먼저 파악한다.
2. **수정 범위 제한**: 버그와 관련 없는 코드는 건드리지 않는다.
3. **3회 실패 시 멈춤**: 같은 접근으로 3번 실패하면 방향을 전환하고 사용자에게 보고한다.

```
[디버깅 보고]
시도한 방법: [1], [2], [3]
모두 실패한 이유: [분석]
제안하는 다른 방향: [대안]
어떻게 진행할까요?
```

## 진행 상황 내레이션

Claude가 모든 자동화 단계를 한국어로 실시간 보고한다.
사용자가 "지금 뭐 하고 있는지" 항상 알 수 있게 한다.

### 내레이션 규칙

1. **단계 시작** 시: "[단계명] 시작합니다 — [한 줄 설명]"
2. **중간 진행** 시: 주요 발견이나 결정 사항을 간결하게 보고
3. **단계 완료** 시: "[단계명] 완료 — [결과 요약]"
4. **선택이 필요할 때**: 옵션을 카드 형태로 제시하고 사용자 선택 대기

### 내레이션 예시

```
[조사 단계] 시작합니다 — 프로젝트 구조와 기술 환경을 파악합니다.

탐색 결과:
- 주 언어: TypeScript
- 프레임워크: Next.js 14 (App Router)
- 테스트: Jest + React Testing Library
- 주요 모듈 3개 발견

[조사 단계] 완료 — 구조 파악이 끝났습니다. 분해 단계로 넘어갑니다.
```

## 로컬 모델 대기 시간 활용

로컬 LLM (ollama 등) 추론은 수 분~수십 분 걸린다.
이 대기 시간을 낭비하지 않는다.

**규칙: 로컬 모델 실행은 항상 `run_in_background: true`로 돌리고, 대기 시간에 다른 작업 수행.**

### 대기 시간 우선순위

| 우선순위 | 작업 |
|---------|------|
| 1 | 다음 실험 코드 미리 준비 |
| 2 | 다른 독립 서브태스크 진행 |
| 3 | 이전 코드 리뷰/리팩토링 |
| 4 | 테스트 코드 작성 |
| 5 | 문서화 / 실험 로그 정리 |
| 6 | 관련 기법 리서치 |

### 자동실험 + 로컬 모델 패턴

```
실험 N 코드 수정 -> background로 테스트 실행
  | 대기 시간 동안
  | 실험 N+1 코드 준비 (가설 + 코드, 실행은 안 함)
  | 또는 다른 독립 서브태스크 진행
  v
실험 N 완료 -> 결과 분석 -> keep/discard
  -> 실험 N+1 즉시 실행 (이미 준비됨)
```

## 세션 관리 & 메모리

긴 세션 = 풍부한 맥락이지만, 컨텍스트 오염 위험.
적정 세션 길이 + 메모리 기록을 병행한다.

### 세션 길이 가이드

| 작업 유형 | 권장 세션 길이 |
|----------|--------------|
| 자동실험 루프 | 실험 5-7개 단위로 세션 교체 |
| 풀사이클 | 분해까지 1세션, 실험부터 새 세션 |
| 단일 버그 수정 | 1세션으로 완료 |
| 코드 리뷰/리팩토링 | 1세션으로 완료 |

### 자동 메모리 기록

세션이 길어지거나 중요한 결과가 나오면 Claude가 자동으로 기록:
- 실험 결과 핵심 요약 -> `memory/experiments.md`
- 아키텍처 결정 -> `memory/architecture.md`
- 발견된 문제점 -> `memory/MEMORY.md`의 Known Issues
- 현재 진행 상태 -> `memory/MEMORY.md`의 Current State

### 세션 교체 시점

Claude가 다음을 감지하면 세션 교체 권장:
- 동일 대화에서 실험 7개 이상
- 컴팩트가 2회 이상 발생
- 작업 주제가 크게 전환
- "아까 그거"의 정확한 참조가 어려워짐

## 상태 저장 경로

모든 Comad Voice 상태는 프로젝트 루트의 `.comad/` 폴더에 저장한다.

```
.comad/
  research/          # 조사 단계 산출물
  plans/             # 분해 결과, 실행 계획
  experiments/       # 실험 로그
  sessions/          # 세션 저장 (T5)
  state.json         # 현재 진행 상태
```

## 토큰 극대화 규칙

1. **AI와 대화하라**: 코드 변경 전후로 "왜?", "다른 방법은?", "이게 최선이야?"
2. **리서치를 아끼지 마라**: 웹 검색 + Agent 탐색으로 충분히 조사
3. **병렬로 돌려라**: Agent tool로 여러 에이전트 동시 실행
4. **Opus를 써라**: 아키텍처/분석은 항상 Opus
5. **실험 해석을 AI에게 맡겨라**: "이 결과를 분석해줘"
6. **대기 시간을 낭비하지 마라**: 테스트는 background, 대기 중에 다음 작업

## 외부 도구 연동 (선택 사항)

Comad Voice는 Claude Code만으로 완전히 동작한다.
아래 도구는 선택적으로 병렬 처리 성능을 높이는 데 사용할 수 있다:

| 도구 | 있으면 | 없으면 |
|------|--------|--------|
| **Codex CLI** | 독립 태스크를 Codex에 병렬 위임 | Agent tool로 Claude 에이전트가 병렬 처리 |
| **tmux** | Codex 병렬 실행에 필요 | Agent tool 기반 병렬만 사용 |
